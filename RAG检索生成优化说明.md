# RAG 检索与生成优化说明

本文档说明当前项目中 RAG 流程的两个关键优化方向：检索前的多查询拆分优化，以及检索后的回答验证与自我纠错优化。其中，多查询拆分优化已经落地到 `backend/services/search_service.py`。

## 1. 优化背景

传统 RAG 流程通常直接使用用户原始问题进行一次向量检索，然后将检索到的片段交给大模型生成回答。这种方式实现简单，但存在两个常见问题：

1. 用户问题可能过短、表达模糊或关键词不完整，导致向量数据库召回不到真正相关的文本片段。
2. 大模型在生成回答时可能没有严格依据检索上下文，出现回答偏离文档、补充不存在事实或产生幻觉的问题。

因此，本次优化围绕 RAG 的两个关键阶段展开：

1. 检索前：扩展用户查询，提高召回率。
2. 检索后：审查生成结果，提高答案可靠性。

## 2. 多查询拆分优化

### 2.1 优化目标

多查询拆分优化的目标是提升向量检索阶段的召回率。系统不再只使用用户原始问题检索，而是先调用大模型生成多个不同角度的查询变体，再分别执行向量检索，最后对结果去重、排序并截取最终 Top K。

例如，用户输入：

```text
怎么报错了？
```

大模型可以扩展为：

```text
系统运行时报错的原因是什么
如何排查程序异常信息
日志中错误提示对应的解决方法
```

这些查询从不同角度覆盖了用户意图，可以提高命中文档中相关 chunk 的概率。

### 2.2 当前实现位置

实现文件：

```text
backend/services/search_service.py
```

新增核心方法：

```python
def _expand_query(self, original_query: str) -> List[str]:
```

该方法会读取环境变量中的 API Key：

```text
ALIYUN_API_KEY
DASHSCOPE_API_KEY
```

如果存在可用 Key，则通过 DashScope OpenAI 兼容接口调用 `qwen-plus`，生成多个查询变体。如果没有 Key 或调用失败，则自动回退为只使用原始查询，保证原检索流程不被中断。

### 2.3 处理流程

当前检索流程如下：

1. 接收用户原始 query。
2. 调用 `_expand_query(query)` 生成查询列表。
3. 对每个扩展 query 分别创建 embedding。
4. 使用 Chroma collection 执行向量检索。
5. 使用 chunk id 对多个查询返回的结果去重。
6. 按相似度 score 从高到低重新排序。
7. 截取最终 `top_k` 个结果返回。

### 2.4 去重策略

多查询检索可能会从不同查询中命中同一个文本片段。为了避免重复上下文影响生成质量，系统使用 chunk id 进行去重：

```python
seen_chunk_ids = set()
```

当某个结果的 chunk id 已经出现过时，后续重复命中的结果会被跳过。

### 2.5 排序策略

所有查询的检索结果会先合并到同一个列表中，然后统一按照 score 降序排序：

```python
processed_results.sort(key=lambda item: item["score"], reverse=True)
processed_results = processed_results[:top_k]
```

这样可以保证最终返回的是多查询合并后的全局 Top K，而不是每个查询各自的局部 Top K。

### 2.6 容错机制

该优化不会强依赖大模型扩展能力。如果调用扩展查询失败，系统会记录日志并回退：

```python
return [original_query]
```

因此，即使 API Key 未配置、网络异常或模型调用失败，原始搜索功能仍然可以正常工作。

## 3. 回答验证与自我纠错优化

### 3.1 优化目标

回答验证与自我纠错用于解决 RAG 生成阶段的幻觉问题。初始回答生成完成后，系统可以再次调用大模型，让其根据用户问题、检索上下文和回答草案进行审查。

审查目标包括：

1. 回答是否严格基于上下文。
2. 是否编造了上下文中不存在的信息。
3. 是否遗漏了用户问题中的关键点。
4. 是否需要重写为更严谨的答案。

### 3.2 建议实现位置

建议在以下文件中实现：

```text
backend/services/generation_service.py
```

可以在 `GenerationService` 类中新增方法：

```python
def _critique_and_correct(
    self,
    query: str,
    context: str,
    draft_response: str,
    provider: str,
    model_name: str,
    api_key: str,
) -> str:
```

### 3.3 建议处理流程

1. 正常根据检索结果生成初稿回答。
2. 将用户问题、检索上下文和初稿回答组成审查 prompt。
3. 调用同一 provider 的大模型进行审查。
4. 如果模型输出 `[PASS]`，直接返回原回答。
5. 如果模型输出 `[CORRECTED]`，提取修正后的回答并返回。
6. 如果审查调用失败，回退返回原回答。

### 3.4 预期效果

该机制可以提升回答的事实一致性，尤其适用于课程问答、文档问答、规章制度查询等对准确性要求较高的场景。

不过，该方案会额外增加一次大模型调用，因此会带来更高延迟和 API 成本。建议将其设计为可配置能力，例如通过参数或环境变量控制是否启用。

## 4. 优化收益

### 4.1 检索召回率提升

多查询拆分可以缓解用户问题表达不完整的问题。相比单查询检索，它更容易覆盖同义词、近义表达和不同问题角度，从而提升相关 chunk 被召回的概率。

### 4.2 上下文质量提升

通过 chunk id 去重和全局 score 排序，最终进入生成阶段的上下文更集中、更少重复，有助于减少无关信息对回答的干扰。

### 4.3 回答可靠性提升

自我纠错机制可以在最终返回前进行一次事实审查，降低模型脱离上下文自由发挥的概率。

## 5. 后续建议

1. 将多查询扩展数量设计为可配置参数，例如默认生成 3 个变体。
2. 增加开关控制是否启用多查询扩展，便于对比单查询和多查询效果。
3. 在搜索结果中记录命中的 expanded query，方便调试和评估。
4. 为回答自我纠错增加独立开关，避免所有请求都承担额外延迟。
5. 增加评估脚本，对比优化前后的召回率、答案准确率和响应时间。

## 6. 总结

本次优化将 RAG 流程从“单次检索 + 直接生成”升级为更稳健的“查询扩展 + 多路检索 + 去重排序”的检索方式。该改动在不破坏原接口的前提下增强了召回能力，并为后续加入回答验证与自我纠错机制打下基础。
